# Internationalization and localization[¶](https://docs.djangoproject.com/en/5.0/topics/i18n/#internationalization-and-localization "Link to this heading")
## Overview[¶](https://docs.djangoproject.com/en/5.0/topics/i18n/#overview "Link to this heading")
The goal of internationalization and localization is to allow a single web application to offer its content in languages and formats tailored to the audience.
Django has full support for [translation of text](https://docs.djangoproject.com/en/5.0/topics/i18n/translation/), [formatting of dates, times and numbers](https://docs.djangoproject.com/en/5.0/topics/i18n/formatting/), and [time zones](https://docs.djangoproject.com/en/5.0/topics/i18n/timezones/).
Essentially, Django does two things:
  * It allows developers and template authors to specify which parts of their apps should be translated or formatted for local languages and cultures.
  * It uses these hooks to localize web apps for particular users according to their preferences.


Translation depends on the target language, and formatting usually depends on the target country. This information is provided by browsers in the `Accept-Language` header. However, the time zone isn’t readily available.
## Definitions[¶](https://docs.djangoproject.com/en/5.0/topics/i18n/#definitions "Link to this heading")
The words “internationalization” and “localization” often cause confusion; here’s a simplified definition:

internationalization[¶](https://docs.djangoproject.com/en/5.0/topics/i18n/#term-internationalization "Link to this term")

Preparing the software for localization. Usually done by developers.

localization[¶](https://docs.djangoproject.com/en/5.0/topics/i18n/#term-localization "Link to this term")

Writing the translations and local formats. Usually done by translators.

Warning
Translation is controlled by the [`USE_I18N`](https://docs.djangoproject.com/en/5.0/ref/settings/#std-setting-USE_I18N) setting. However, it involves internationalization and localization. The name of the setting is an unfortunate result of Django’s history.
Here are some other terms that will help us to handle a common language:

locale name[¶](https://docs.djangoproject.com/en/5.0/topics/i18n/#term-locale-name "Link to this term")

A locale name, either a language specification of the form `ll` or a combined language and country specification of the form `ll_CC`. Examples: `it`, `de_AT`, `es`, `pt_BR`, `sr_Latn`. The language part is always in lowercase. The country part is in titlecase if it has more than 2 characters, otherwise it’s in uppercase. The separator is an underscore.

language code[¶](https://docs.djangoproject.com/en/5.0/topics/i18n/#term-language-code "Link to this term")

Represents the name of a language. Browsers send the names of the languages they accept in the `Accept-Language` HTTP header using this format. Examples: `it`, `de-at`, `es`, `pt-br`. Language codes are generally represented in lowercase, but the HTTP `Accept-Language` header is case-insensitive. The separator is a dash.

message file[¶](https://docs.djangoproject.com/en/5.0/topics/i18n/#term-message-file "Link to this term")

A message file is a plain-text file, representing a single language, that contains all available [translation strings](https://docs.djangoproject.com/en/5.0/topics/i18n/#term-translation-string) and how they should be represented in the given language. Message files have a `.po` file extension.

translation string[¶](https://docs.djangoproject.com/en/5.0/topics/i18n/#term-translation-string "Link to this term")

A literal that can be translated.

format file[¶](https://docs.djangoproject.com/en/5.0/topics/i18n/#term-format-file "Link to this term")

A format file is a Python module that defines the data formats for a given locale. Previous page and next page
